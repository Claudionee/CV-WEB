Observabilidad · Seguridad · Multi-Tenant

Plataforma centralizada de auditoría y monitoreo de logs

Implementación interna de Graylog como solución unificada para la auditoría de accesos y monitoreo de eventos en una cartera de clientes corporativos distribuidos en entornos híbridos Huawei Cloud y AWS.

Huawei Cloud AWS Linux Compliance SIEM-ready
Contexto

La operación gestionaba más de 400 servidores repartidos en múltiples nubes y clientes, con logs dispersos en cada sistema y sin un punto único de auditoría. Las investigaciones de incidentes requerían accesos SSH manuales servidor por servidor, y no existía capacidad de detección proactiva de accesos anómalos ni trazabilidad centralizada para auditorías.

Enfoque técnico

Se diseñó una plataforma Graylog en alta disponibilidad con ingesta vía Syslog, GELF y filebeats. Separación lógica por cliente mediante streams y permisos granulares, dashboards por dominio de servicio y pipelines de enriquecimiento para normalizar eventos heterogéneos (Linux, Windows, aplicaciones SAP, firewalls).

Arquitectura de referencia
// fuentes de eventosLinux SUSE/RedHat · Windows Server · SAP Business One · FortiGate · aplicaciones corporativas
▼ syslog / gelf / beats
// capa de ingestaGraylog inputs (TCP/UDP 514, GELF 12201) + pipelines de parsing y enriquecimiento
▼
// capa de procesamientonodos Graylog en HA · backend OpenSearch/Elasticsearch · MongoDB para metadatos
▼
// capa de consumostreams por cliente · dashboards por dominio · alertas vía email/Teams · exportación a SIEM externo
Stack
Graylog OpenSearch MongoDB Syslog GELF Filebeat Linux Ansible
Decisiones clave
  • Multi-tenancy por streams en lugar de instancias separadas: menor costo de mantenimiento y rollouts rápidos.
  • Pipelines de normalización para convertir eventos heterogéneos en un esquema común, habilitando correlación cross-cliente.
  • Retención por capas (hot / warm / archive) alineada con obligaciones de auditoría y costo de almacenamiento.
  • Ansible como vehículo de despliegue de los agentes en flota, evitando configuración manual por host.

Migración Multi-Cloud · SAP HANA · 6R

Migración de SAP HANA desde On-Premise hacia Huawei Cloud

Diseño y ejecución de la migración completa de entornos SAP Business One sobre HANA desde infraestructura on-premise hacia una landing zone en Huawei Cloud, aplicando metodología 6R y minimizando la ventana de cutover.

SAP HANA Huawei Cloud SUSE SLES 6R methodology Cutover planificado
Contexto

El cliente operaba su ERP sobre SAP Business One con base de datos HANA en un datacenter on-premise con hardware llegando a fin de vida útil. Se requería modernizar la plataforma con mínima interrupción del negocio, preservar integridad transaccional y habilitar escalabilidad elástica para cierres de mes y procesos de alto consumo.

Enfoque técnico

Fase de Assessment detallada sobre dependencias de red, integraciones y tamaño real de memoria requerida por HANA. Estrategia combinada del framework 6R: Rehost para servidores aplicativos, Replatform para la capa de base de datos aprovechando instancias optimizadas en memoria de Huawei Cloud. Migración por oleadas con entornos DEV/QA validados antes del cutover productivo.

Arquitectura destino
// landing zoneVPC con subnets públicas/privadas · NAT Gateway · Security Groups segmentados por tier
▼
// frontend / aplicativoECS con SAP B1 sobre SLES · ELB frontal · certificados TLS gestionados
▼
// base de datosECS memory-optimized con SAP HANA · EVS de alto IOPS · snapshots periódicos
▼
// respaldo y DRCBR para backups programados · OBS para retención de largo plazo · cross-region opcional
Stack
Huawei Cloud SAP HANA SAP Business One SUSE SLES SMS DRS OMS CBR OBS Terraform Ansible
Decisiones clave
  • Replatform para HANA, no simple rehost: permitió aprovechar instancias memory-optimized y reducir latencias.
  • Migración por oleadas con entornos DEV/QA en la nube antes del cutover productivo, reduciendo sorpresas el día D.
  • SMS + DRS en paralelo para cómputo y base de datos, con replicación continua hasta minutos antes del corte.
  • Hardening SUSE/SLES estandarizado vía Ansible para asegurar cumplimiento de políticas y notas SAP.
  • Runbook de cutover ensayado en al menos dos simulacros previos al evento productivo.

High Availability · AWS · Multi-AZ

Arquitectura High-Availability en AWS para aplicación crítica

Re-architecting de una aplicación mono-servidor hacia un diseño Multi-AZ con escalado automático, almacenamiento compartido resiliente y recuperación transparente ante fallos de infraestructura.

AWS Multi-AZ Auto Scaling Re-architecting HA
Contexto

La aplicación corría sobre un único EC2 con base de datos local, sin redundancia ni capacidad de escalar. Una caída del host o de la zona de disponibilidad significaba indisponibilidad total del servicio y recuperación manual. El objetivo fue alcanzar alta disponibilidad y escalabilidad sin rediseñar la lógica de la aplicación.

Enfoque técnico

Desacople de cómputo, datos y almacenamiento: frontend detrás de un Application Load Balancer, nodos EC2 dentro de un Auto Scaling Group repartidos en múltiples AZ, base de datos migrada a RDS Multi-AZ y archivos compartidos movidos a EFS. Todo monitoreado por CloudWatch con auto-recovery y backups automatizados hacia S3 con políticas de ciclo de vida a Glacier.

Arquitectura de referencia
// route 53DNS con health checks · failover automático
▼
// application load balancerdistribución entre AZs · SSL terminado · WAF opcional
▼
// auto scaling groupEC2 en AZ-a y AZ-b · launch template versionado · escalado por CPU/memoria
▼
// datos & storageRDS Multi-AZ con réplicas de lectura · EFS compartido entre instancias · S3 + Glacier para backups
▼
// observabilidadCloudWatch metrics y logs · alertas SNS · auto-recovery ante fallos de instancia
Stack
AWS EC2 Auto Scaling ALB RDS Multi-AZ EFS S3 + Glacier CloudWatch Route 53 IAM Terraform
Decisiones clave
  • RDS Multi-AZ antes que réplicas manuales: reduce operaciones y garantiza failover automático.
  • EFS en lugar de EBS compartido: habilita escritura concurrente desde múltiples instancias sin locks.
  • Launch templates versionados para poder hacer rollback controlado de imágenes.
  • Lifecycle policies S3 → Glacier para bajar el costo de retención larga sin perder compliance.
  • Health checks a nivel de aplicación, no solo de red, para detectar degradación real del servicio.
Resultados
HA

Eliminación de puntos únicos de falla en el diseño

Infrastructure as Code · Automatización

Framework de IaC con Terraform y Ansible en entornos Multi-Cloud

Estandarización del aprovisionamiento de infraestructura y configuración de sistemas operativos mediante un framework de IaC reutilizable, aplicable tanto a Huawei Cloud como a AWS, con el objetivo de eliminar drift y reducir tiempos de despliegue.

Terraform Ansible IaC Multi-Cloud DevOps
Contexto

Los despliegues de infraestructura se realizaban mayoritariamente de forma manual desde consolas cloud, lo que generaba inconsistencias entre entornos, ausencia de trazabilidad de cambios, tiempos largos de aprovisionamiento y dependencia de conocimiento tácito del operador.

Enfoque técnico

Diseño de un framework basado en dos capas: Terraform para todo lo que es infraestructura declarativa (redes, cómputo, storage, IAM) y Ansible para configuración idempotente de los sistemas operativos y aplicaciones sobre ellos (hardening, instalación SAP, agentes de monitoreo, usuarios y permisos).

Estructura del framework
// terraform · módulos reutilizablesvpc · compute · database · security · backup · observability · por cada provider
▼
// terraform · composiciones por proyectocombinan módulos para cada cliente/entorno · variables por ambiente (dev/qa/prod)
▼
// remote stateOBS para proyectos Huawei · S3 para proyectos AWS · state locking habilitado
▼
// ansible · roleshardening-linux · hardening-windows · sap-prereqs · monitoring-agent · patching
▼
// orquestaciónpipelines con plan → apply · aprobaciones manuales en prod · Ansible Vault para secretos
Stack
Terraform Ansible Git CI/CD Huawei Cloud AWS Ansible Vault Python Bash
Decisiones clave
  • Módulos pequeños y específicos por recurso, componibles según necesidad: más mantenibles que mega-módulos.
  • Separación de infraestructura y configuración: Terraform crea la caja, Ansible configura lo que vive adentro.
  • Remote state con locking obligatorio: evita corrupción de estado por ejecuciones concurrentes.
  • Ansible Vault para todo lo que sea secreto, nunca credenciales en claro en el repo.
  • Pipelines con plan previo y aprobación manual en producción: automatización sin perder control.

Backup · DR · Continuidad Operativa

Estrategia de Backup & Disaster Recovery Multi-Cloud

Diseño y aplicación de una estrategia unificada de respaldo y recuperación ante desastres para cargas críticas distribuidas en Huawei Cloud y AWS, combinando servicios nativos y soluciones de terceros para cumplir RTO/RPO diferenciados por criticidad de workload.

DR Backup Multi-Cloud RTO/RPO Runbook
Contexto

Diversos clientes operaban sin una estrategia formal de DR: backups eran tomados ad-hoc, sin pruebas periódicas de restauración ni documentación clara de los procedimientos a ejecutar ante un incidente mayor. Recuperarse dependía del conocimiento individual del ingeniero de turno.

Enfoque técnico

Clasificación de workloads por criticidad en 3 tiers, con objetivos RTO/RPO diferenciados por tier. Aplicación del patrón 3-2-1 (tres copias, dos medios distintos, una offsite) con servicios nativos de cada cloud y Acronis como capa adicional para workloads específicos. Runbooks de DR documentados y ensayados.

Modelo de tiering
// tier 1 · misión críticaSAP HANA y core bancario/financiero · RPO minutos · RTO horas · replicación cross-region + snapshots frecuentes
// tier 2 · importanteaplicaciones de negocio · RPO horas · RTO medio día · backups diarios + retención mensual
// tier 3 · estándarentornos no productivos · RPO diario · RTO best-effort · lifecycle hacia almacenamiento frío
Stack
Huawei CBR AWS Backup OBS S3 Glacier Acronis Snapshots EVS/EBS Bash Python
Decisiones clave
  • Tiering antes que "backup para todo igual": permite invertir donde realmente importa y optimizar costos.
  • Lifecycle automatizado hacia almacenamiento frío (Glacier / OBS Archive) para cumplir retención larga barata.
  • Pruebas de restore periódicas: un backup no probado no es un backup, es una esperanza.
  • Runbooks versionados en Git junto con el IaC, no en wikis inconexas.
  • Cross-region solo para Tier 1: su costo se justifica solo en workloads que no pueden caer.

RTO/RPO

3-2-1
